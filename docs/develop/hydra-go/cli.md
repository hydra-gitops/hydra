# CLI Architecture

The CLI module (`cli/`) implements the Hydra command-line interface using the Cobra framework. It follows a strict separation: Cobra commands define the user interface, action handlers contain the orchestration logic, and core functions provide the business logic.

## Key Concepts

- **Command hierarchy** - Root command `hydra` with top-level groups `ci`, `version`, `local`, `argocd`, and `cluster`. `hydra ci` contains `config`, `secrets`, and `run`; `hydra ci run` contains `download`, `auto`, `test`, `release`, `promote`, `publish`, `verify` (`validate` as deprecated alias), `sprint`, `upgrade`, `sync`, and `update`. `hydra local` contains `find`, `config`, `template`, `list`, `source`, `values`, `refs`, `inspect`, `review`, `test`, and `export`. `hydra argocd` contains the read-only `status` command plus the canonical `sync` family (`auto`, `manual`, `prevent`). `hydra gitops` contains `validate-current-context`, `dump`, `list`, `refs`, `apply`, `diff`, `template`, `values`, `status`, `review` (`app`, `cluster`), `system`, `untracked`, `uninstall`, `cert-manager`, `backup`, `sync` (`status`, `auto`, `manual`, `prevent`), `scale` (`up`, `down`, `status`), and `inspect` (`clusterTreeSubcommandEnabled` in `cli/cmd/cluster.go` registers `hydra gitops inspect`)
- **Refs listing (transitive)** - `hydra local refs <cluster> <id>` and **`hydra gitops refs <cluster> <id>`** print **transitive** reference reachability from the anchor id (BFS, **max 10** levels per direction) with **signed distance** aligned to Hydra UI: **0** = anchor, **positive** = outgoing hops, **negative** = incoming hops. Output is an **entity-centered** list with distance, direction, and relation/label fields—not a legacy “only edges incident on `<id>`” slice. No flag toggles transitivity; it is always enabled.
- **Inspect command (TUI)** - `hydra local inspect <cluster> [id]` opens a Bubble Tea UI. `LoadLocalTreeGraph` runs once before the TUI loop (`core/commands/cluster_refs_tree.go`); the handler reuses `LocalTreeGraph` for the whole session so picker, graph view, and Esc-back-to-picker do not re-render templates. Without `id`, `LocalTreeGraph.CandidateIds` feeds an id picker (`cli/tui/id_picker.go`) with a `/`-opened overlay filter dialog, a text input focused by default, a field dropdown reachable via `Tab`, shared filter fields (`id`, `name`, `gvk`, `gvkn`, `group`, `version`, `kind`, `namespace`), toggling sort direction on `s` (ascending → descending for the current column, then advancing to the next column; **Shift+S** reverses that cycle), a top help line (between the picker header and the list, not as a footer), and **no Status column**. With `id`, or after picking, the reference graph runs in `cli/tui/ref_tree.go`: **Incoming refs** + **Outgoing refs** use **transitive** BFS expansion (**max 10** levels), **`Dist`** is the **first** list column, the **current entity** appears as its own list row at **Dist 0**, `/` opens the same style of filter dialog over the graph, and `s` toggles the sort direction (ascending ▲ → descending ▼) for the current column, then advances to the next column (default **Dist** ascending; full cycle follows list column order Dist → Ref id → Relation → Status: dist↑ → dist↓ → id↑ → id↓ → relation↑ → relation↓ → status↑ → status↓ → dist↑ …); **Shift+S** reverses the cycle (descending → ascending on the same column, or previous column descending); the active column header displays ▲ or ▼ and inactive columns reserve a space in the same slot so headers do not shift. In **local inspect**, picker/header/list omit **Status** entirely. In **cluster inspect**, `LoadClusterTreeGraph` is used once per invocation (same file), then `ClusterTreeGraph.CandidateIds` and `PickerRowStatuses` provide the picker rows, and `RunRefTree` runs with **Status** enabled in picker, header, and list (`ok` / `missing` / `cluster only`) based on merged template-vs-cluster edges. In both modes, `EnsureStartId` and `RunRefTree` use the cached graph’s entities and `Refs`; `↑`/`↓` moves through visible rows with automatic scroll-to-selection, **PgUp**/**PgDn** scroll the list by one page, Enter follows an edge, Escape closes the active popup or goes back, and at the root entity with no history Escape returns to the id picker when the command was started without `<id>` on the CLI; q quits
- **Inspect TUI — detail panel (specification)** - `hydra local inspect` and `hydra gitops inspect` share `RunRefTree` / `RefTreeModel` in `cli/tui/ref_tree.go`; this specification is the contract for how vertical space, filtering, sorting, and detail rendering behave (aligned with `hydra/docs/manual/cli/shared/hydra-tree.md`). **Structure:** entity header, bordered **list** panel (incoming + outgoing), bordered **detail** panel (merged `Ref` for the selected row), optional filter popup, top help line (between entity header and list panel border, not as a footer). **Column headers:** The currently sorted column header displays a ▲ (ascending) or ▼ (descending) direction indicator; other columns show a space in that slot so the header row width stays stable. The header row is **sticky**: list vertical scroll applies only to rows below it. **Sizing input:** `WindowSizeMsg` drives `width`/`height`; `computeLayout` (and `panelStyle` frame sizes) derive inner list lines and inner detail lines from the remaining vertical budget after header, top help line, gaps, and borders. **Detail height:** The layout **must** give the detail panel enough inner height to show the **full** word-wrapped detail for the **current** selection **whenever** the terminal budget allows—unused vertical space **must not** be withheld in a way that truncates detail lines that could have fit. **Short terminal:** If minimum usable list height plus full detail still exceeds the budget, **cap the list first** (down to a defined minimum), assign the rest to the detail, and if wrapped detail still exceeds visible lines, keep a **detail vertical scroll offset** over logical detail lines. **Filter popup:** `/` opens a modal overlay in both picker and graph; the text input is focused first, `Tab` and `Shift+Tab` switch to or from the field dropdown, `Enter` applies, and `Escape` closes the popup before any navigation exit behavior. **Selection stability:** Changing the active filter or sort mode recomputes the visible rows, clamps cursor/list scroll to the new visible set, and keeps the detail pane aligned to the highlighted visible row. **Input routing:** **PgUp**/**PgDn** page the **list** only; **separate** key bindings scroll the detail pane when it overflows; those bindings **must** appear in the top help line and match the user manual’s help-line description. **Selection change:** Replacing the selection **must** reset detail scroll to the top of the new ref’s text. **Unit tests** must cover at least: picker popup opening and focus defaults, picker field-specific filtering and sort cycling, graph popup opening and filtering against visible rows, graph sort cycling, help text rendered above the panel (not as a footer) for `/` and `s`, sort direction toggle cycling (ascending → descending → next column), sort direction indicator (▲/▼) next to the active column header, plus the existing detail-height/scroll guarantees in `cli/tui/ref_tree_test.go`. Similar tests should exist for the id picker in `cli/tui/id_picker.go`: sort toggle cycling (ascending → descending → next column), help text position above the panel, and sort direction indicator in column headers
- **Review command families** - `hydra local review` is the flat local render-only review under `hydra local` (no `refs` subcommand); `hydra gitops review` groups **`app`** (app id patterns, like the historical top-level review) and **`cluster`** (repository cluster name, no `.`) under `hydra gitops` (no `refs` subcommand)
- **Selected-app render namespace invariant** - Commands that render a subset of apps (`hydra local template`, `hydra local find`, `hydra local review`, `hydra gitops review`, and similar) must apply the same namespaced-resource namespace propagation as full `RenderCluster` so omitted `metadata.namespace` is filled from the app namespace before ref validation or `Entity.Id()`-based indexing. See `commands.md` (`RenderClusterSelectedApps`, `ReviewRefs`) and `references.md`
- **Local review scope** - `hydra local review <appId...> [--exclude-app ...]` renders the **selected** apps for **source** entities (same namespace normalization contract as other selected-app renders) and builds a **separate target index** from the templates of **all** effectively enabled apps on the **same** cluster derived from those `appId` arguments. **Upstream Kubernetes bootstrap** entities (core defaults plus RBAC bootstrap names; see `references.md`) are **merged** into that template target set so references to built-in `ClusterRole` objects and similar defaults do not false-positive as `missing target resource`. **Per-namespace kubernetes-defaults** (`ServiceAccount/default`, `ConfigMap/kube-root-ca.crt` with synthetic `ca.crt`, `Namespace`) are merged for every namespace present in that template target set (deduped like `RenderCluster`). It shares core review logic with cluster review and differs by target provenance (local full-cluster template render vs live API). `--exclude-app` limits which apps contribute **sources** and findings, not which apps contribute **targets**. Apps with `enabled: false` are omitted from both source and target template sets. Explicit Secret/ConfigMap key selections stay as repeated relation attributes (for example `type=key`), findings are reported only for selected-app sources, repeated hits with identical target and message are grouped, missing target objects use the finding text `missing target resource` (direct object or a ref declaring **`"origin:generated": job`** / **`"origin:generated": controller`** such as `SopsSecret` secret templates must be considered), and the command returns a non-zero exit code when any finding exists. Grouping and sorting may still operate on the full finding set in memory; only the **stdout write path** is progressive: after that set is final, each finding is emitted in order. **Default stdout** is a short human-readable text block per finding (with optional ANSI styling). With **`--yaml`**, each finding is emitted as its own YAML sequence element, without marshaling the entire slice as one value at the end (legacy machine-oriented format). Coloring for both modes follows the shared `ColorFlag` surface (`--color`, `--no-color`, `--color-mode`): default is TTY auto-detect, not forced-on color. Debug-level logs mark slow post-render phases (ref-parser collection, `references.Refs`, explicit key attribute enrichment, target-key normalization, grouping and sorting) so operators can profile stalls without changing success output. After those phases, local review also evaluates **template vs `uninstall` / `uninstall-force` ref ownership**, **excluding** groups tagged **`runtime`**. **`hydra gitops review`** runs the same **template** ownership pass (still **excluding** **`runtime`**), then evaluates **live cluster resources whose ids appear in no standalone template render** using the full predicate set **including** **`runtime`** (same as cluster uninstall for non-template resources). **`hydra gitops uninstall`** assignment still includes **`runtime`** predicates for all cluster objects (unchanged).
- **Cluster review scope** - **`hydra gitops review app <appId...>`** and **`hydra gitops review cluster <cluster>`** (`--exclude-app` on both) use the same review pipeline: **sources** from the locally rendered resolved app set, **targets** from **all** live entities on the cluster, plus an **auxiliary** per-namespace default bundle derived from the **full enabled-app template render** for **same-namespace** references to `ServiceAccount/default` or `ConfigMap/kube-root-ca.crt` only (see `references.md`). The **`cluster`** subcommand resolves apps via `Cluster.AppIds` for the named repository cluster (then applies excludes); **`app`** uses `ResolveAppIdsFromConfig`. Only **`hydra gitops review cluster`** runs a post-pass that **audits** the same upstream bootstrap catalog against the live inventory (server minor from discovery), emitting **`missing cluster default resource`** for absent expected IDs and suppressing duplicate `missing target resource` for those IDs. **`hydra gitops review app`** skips that full-catalog audit. If the server version cannot be read, the bootstrap audit is skipped while auxiliary namespace defaults still apply. `--exclude-app` limits which apps contribute **sources** and findings, not the breadth of the live target inventory. Apps with `enabled: false` are not rendered by Hydra and therefore supply **no sources**; the live **target** inventory remains complete. It preserves the same repeated key attributes, grouped finding behavior, `missing target resource` text when neither a live target nor an in-set declared materialization (`"origin:generated": job` / `"origin:generated": controller`) exists, non-zero exit on findings, the same grouping-then-sequential-emission stdout contract (default human text per finding, optional `--yaml` for YAML sequence elements), color flags with TTY auto-detect by default, and the same debug instrumentation boundaries after templating. Ref ownership uses **two** passes: template ids exclude **`runtime`**-tagged uninstall groups (same as local review); live objects with **no** template id are checked with **`runtime`** predicates included, and ambiguous multi-app matches emit **`RefOwnershipAmbiguousClusterOnlyFinding`**. **`ref ownership: cluster-only resource has no Hydra app assignment`** is emitted only for **`hydra gitops review cluster`** (`AppendRefOwnershipReviewFindings` with `reportUnassignedClusterOnlyResources: true`).
- **ArgoCD status vs cluster status** - `hydra argocd status [appId...] [--exclude-app ...]` is the ArgoCD-facing read-only command and must show the real application sync state reported by ArgoCD, while `hydra gitops status <appId...> [--exclude-app ...]` is the cluster-based per-app in-sync check derived from Hydra render output versus tracked live cluster resources
- **ArgoCD sync family** - `hydra argocd sync <auto|manual|prevent> <appId...> [--exclude-app ...]` is the canonical sync command family; user-facing docs, help text, synopses, and examples should no longer describe these operations under `hydra gitops`
- **Manual structure** - User docs now use `hydra/docs/manual/README.md` as a root landing page, split CLI docs under `hydra/docs/manual/cli/` (`global`, `local`, `argocd`, `cluster`), document `hydra argocd ...` under `hydra/docs/manual/cli/argocd/`, and place non-CLI user docs alongside it in `charts-repository/`, `gitops-repository/`, and `ui/`
- **Architecture layers** - `cmd/` (Cobra commands, no logic) -> `action/` (orchestration) -> `core/` (business logic)
- **Dependency injection** - Commands accept action functions as parameters for testability
- **Flag system** - Interface-based flag detection via `DefineFlags()`, composable flag structs with embedding
- **Optional user kubeconfig mapping** - Operators may place `$XDG_CONFIG_HOME/hydra/config.yaml` (default `$HOME/.config/hydra/config.yaml`) with a `contexts` list mapping absolute cluster directory paths to a kubeconfig file plus context name. [`HydraClusterAccess`](../../../hydra-go/core/hydra/validation.go) builds `genericclioptions.ConfigFlags` with that mapping before `allowedContexts` validation, so the safety rules apply to the same kube context used for REST. See [details/cli.md](details/cli.md#optional-user-kubeconfig-xdg) and the user manual [configuration.md](../../manual/cli/configuration.md).
- **Helm render cache opt-out** - Commands that register shared flags via `DefineFlags()` expose `--no-cache` to disable persistent on-disk Helm template caching (under each root app’s `.hydra/cache/helm/` in the GitOps tree) and the related in-process values/template/chart caches for that process. Equivalent environment variable: `HYDRA_NO_CACHE` (truthy values such as `1`, `true`, `yes`). CLI wins when both are set.
- **Pager mode** - Global `--less` (handled in `ExecuteArgs` before Cobra) re-executes Hydra without `--less`, prepends `--color` / `--color-log` for the subprocess when those outputs are not explicitly disabled, merges stdout and stderr into one pipe, and runs `$PAGER` (default `less -SR +G`) in the foreground with the process environment unchanged
- **FlagBuilder** - Generic type-safe builder for flag definition with validation hooks and shell completion
- **Backup command secret filters** - `hydra gitops backup create` and `hydra gitops backup restore` support `--include` / `--exclude` to narrow the secrets selected by app-defined backup discovery; `backup create` fails for ownership-invalid target namespaces, and `backup restore` reports those backups as `skipped`; `backup restore` additionally supports `--create-namespaces` to prepare missing target namespaces for the already selected ownership-valid backups
- **Backup selection scope** - `backup list`, `backup restore`, and `backup diff` discover backup manifests only from the selected app IDs; namespace allowlists and `--all` are not part of backup selection
- **Apply-integrated restore** - `hydra gitops apply` runs backup restore as a normal apply phase for backup manifests rendered from the selected app IDs. Earlier namespace application in the same phase plan may still prepare selected app namespaces before that restore phase runs. Apply logging should stay phase-oriented, avoid duplicate generic `applying N resources` messages for the same operation, and treat the `Backup overview` as the restore summary instead of preceding it with extra per-secret `up-to-date` lines. In `--bootstrap` mode, ordinary selected-app `SopsSecret` CRs still follow the normal apply path, while ownership-invalid backup resources must not later be applied as ordinary resources
- **Bootstrap guard** - After render, `hydra gitops apply` evaluates `global.hydra.refs` groups tagged `bootstrap-guard` against rendered template entities before bootstrap secret conversion. Matching resources without `--bootstrap` or `--skip-bootstrap-guard` abort the command; `--bootstrap` and `--skip-bootstrap-guard` are mutually exclusive
- **CI subcommands** - `test`, `release`, `promote`, `publish`, `validate`, `sprint`, `upgrade`, `sync`, and `update` are wired via the shared `newCiSubcommand` helper; `config` is the interactive exception with its own Cobra command because it takes `path` plus `--no-color`
- **Local query workflow** - `hydra local find` resolves AppId patterns from the filesystem, renders matching apps locally, filters resources with CEL predicates, projects values with a required CEL `--pick` expression, optionally deduplicates projected values with `--uniq`, and prints a YAML array
