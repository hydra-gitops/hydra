# Business Commands Architecture

The commands package (`core/commands/`) contains high-level orchestration functions for cluster operations. These functions are called by CLI action handlers and compose lower-level operations (entity processing, Helm rendering, Kubernetes API calls) into coherent workflows.

## Key Concepts

- **RenderCluster** - Renders all (or selected) apps in a cluster to entities with scope info and namespace propagation
- **Cluster apply: CRD scope vs apply eligibility** - `hydra gitops apply` must resolve API scope for custom resources using CustomResourceDefinition manifests from **all** apps rendered for the target cluster (full cluster render), not only from the app IDs selected for the current command, so that scope metadata (for example namespaced vs cluster-scoped) can come from a CRD shipped with a different chart than the custom resource instance. Separately, before mutating the cluster, apply must enforce that every CRD required to admit resources in the **selected** apply set is satisfiable: the CRD must either already exist (and be established) on the target cluster **or** be present among the CRD objects included in the manifests for the selected apps for this run. If a custom resource in the selection depends on a CRD that appears only in a non-selected app’s render and that CRD is not live on the cluster, the command must abort with an error (default `--crd-mode error`). This avoids applying instances whose definitions would not be installed by the selected set
- **RenderClusterSelectedApps** - Renders a selected AppId set into entities enriched with `appIds`, `appNamespace`, and source paths; used for local query workflows (`hydra local find`), reference review (`ReviewRefs`), and similar selected-app flows. It must apply the same scope-map and namespace propagation as `RenderCluster` (`ApplyScopeInfoMaps`): for namespaced GVKs, missing `metadata.namespace` on the template `Unstructured` is filled from the app namespace before consumers run ref logic or index by `Entity.Id()`. Internally, the early selected-app path may use a dedicated non-CLI `CrdMode` branch to keep still-unknown GVKs unchanged until fuller scope information is available.
- **ReviewRefs** - `hydra local review` and `hydra gitops review` share one orchestration path: normalize namespaced sources per the selected-app render contract, build resource-level refs with explicit Secret/ConfigMap keys as repeated `type=key` attributes, validate targets (including **recursive** expansion of `"origin:generated": job` / `"origin:generated": controller` materializations to a **fixpoint**—see [references.md details](details/references.md#generated-targets-recursive-evaluation-and-fixpoint)), group equivalent target+message pairs, and fail when any finding exists. **Sources** differ by mode: **local review** uses only template entities from the apps selected for the review (after `appId` pattern resolution and `--exclude-app`); **cluster review** builds a candidate set from rendered template entities **and** clone materializations of the selected apps, applies API version normalization so template IDs match cluster IDs despite potential version differences, then **filters** that candidate set to only include resources whose IDs also exist on the live cluster (`ListClusterAll`), keyed as `KeyTemplateEntity`—resources that are rendered but not yet deployed are not reviewed, but references from reviewed sources to non-existent targets (for example a deployed Deployment referencing a missing Secret) are still reported. In both modes, refs are evaluated only for sources, and findings may list only such sources. **Targets** differ by mode: **cluster review** resolves targets against live cluster entities only (`ListClusterAll`, keyed as `KeyClusterEntity`), **without** clone materializations in the target set; **local review** resolves targets against the rendered templates of **all** effectively enabled applications on the **same** cluster as determined from the selected `appId` set (the existing single-cluster constraint for review). The inner shared review function `reviewRefsEntitiesWithTargetsAndParserSetsCallback` is unchanged and serves as the shared core for both local and cluster review; ref-parsers remain unchanged (from selected apps + cluster ConfigMaps). `--exclude-app` narrows which apps contribute **source** entities and therefore which apps can appear in findings; it must **not** shrink the loaded **target** entity set in either mode. Apps with `enabled: false` are not rendered by Hydra, so they contribute **no reference sources**. They are omitted from the **local** template **target** set; **cluster** review still resolves **targets** against the **full** live cluster inventory regardless of `enabled`, because that inventory reflects the API, not the current render selection. A referenced `Secret` or `ConfigMap` counts as present if the target set contains a matching object **or** if **ref-parser-declared** edges with **`"origin:generated": job`** or **`"origin:generated": controller`** (including chained edges after fixpoint stabilization) account for that target id and keys; emit `missing target resource` only when **neither** a direct object **nor** that stabilized materialization exists. Materialization rules come from embedded, Helm, and ConfigMap parsers—not from ad hoc Go-only scans of specific CRDs. Cross-app and cross-namespace references are valid when the target ID exists in the target set. Implementation may still build and sort the **complete** finding list before any stdout write; stdout is filled **sequentially** (one YAML list element per finding, in final order) and must not rely on a single `yaml.Marshal` of the whole array at the end. Shared review logic should emit findings through an injected callback so unit tests can capture each finding directly without replacing process-wide stdout. CLI handlers should create the default callback via a small factory that receives `ColorFlag` and returns a stdout-writing YAML emitter. YAML highlighting therefore still defaults to TTY auto-detect and can be forced on or off with `--color` / `--no-color` (and `color-mode` alongside other YAML commands). Debug logs bracket expensive post-render steps—`HydraAppRefParsers`, `references.Refs`, key-attribute enrichment, target-key normalization, grouping and sorting—so slow slices of the pipeline are visible without altering structured stdout
- **ListCluster** - Lists live cluster resources with CEL predicate filtering and ArgoCD tracking
- **LocalTreeGraph** - Holds refs and entities from one template-only full-cluster pipeline run. The shared render helper is `clusterRefsAllFromTemplates` in `core/commands/cluster_refs_for_id.go`. `hydra local inspect` loads this graph once before the TUI loop (via `LoadLocalTreeGraph` in `core/commands/cluster_refs_tree.go`) and reuses it for `CandidateIds()`, `EnsureStartId()`, and `Refs` access so moving between the id picker, the reference graph, and back does not re-render all Helm templates on every iteration. It mirrors the **`ClusterTreeGraph`** pattern used by `hydra gitops inspect` (`LoadClusterTreeGraph`, in-memory `CandidateIds` / `EnsureStartId` / picker status helpers, ref slice for `RunRefTree`) but without merging or consulting live cluster inventory—template entities and computed refs only
- **Transitive reference listing and inspect navigation (BFS, distance)** - `hydra local refs`, **`hydra gitops refs`**, `hydra local inspect`, and `hydra gitops inspect` resolve **transitive** dependencies over the directed reference graph: breadth-first expansion from the anchor entity, **at most 10 hop levels** per direction, matching the Hydra UI graph convention. **Distance** is entity-centric and signed: **0** is the **current** (anchor) entity; **positive** distances count **outgoing** hops along `From → To` edges; **negative** distances count **incoming** hops (reverse direction). There is **no** CLI flag to disable transitive expansion; it is always on for these commands. **`hydra local refs`** prints an **entity-centered** result set (each row is a reachable peer id with distance, direction, and merged relation/label metadata—not merely a flat dump of direct `Ref` records that touch the anchor). **`hydra gitops refs <cluster> <id>`** is the live-cluster analogue: refs are computed from **merged** template and cluster sources (same merge model as `hydra gitops inspect`), then BFS-labeled for stdout. **`hydra local inspect`** / **`hydra gitops inspect`** show the **current entity as its own row** at **Dist 0** in the TUI list and add a **`Dist`** column on every incoming/outgoing row; list contents are the transitive frontier (still capped at 10 levels), not only immediate neighbors. Shared graph construction for listing vs inspect should stay consistent with the existing `LocalTreeGraph` / `ClusterTreeGraph` single-load-per-invocation contract
- **ArgoCD status surface** - `hydra argocd status` is the read-only ArgoCD-facing status command and reports the real application sync state from ArgoCD rather than Hydra-computed drift; without app IDs it considers all visible Applications, after which `--exclude-app` patterns may subtract from that set
- **Cluster status surface** - `hydra gitops status <appId...>` is the cluster-based per-app in-sync check and must derive its result from rendered desired resources compared with tracked live cluster resources
- **ArgoCD sync surface** - `hydra argocd sync <auto|manual|prevent> <appId...>` is the canonical command family for sync changes; multi-app variants document `--exclude-app` to subtract apps from the resolved target set
- **Backup** - Per-app backup/restore of Kubernetes secrets as SOPS-encrypted SopsSecret CRDs using `backup` tag in `global.hydra.refs`. `backup create` and `backup restore` support secret-level `--include` / `--exclude`; `backup restore` additionally supports `--create-namespaces` to create missing target namespaces for the already selected backup secrets. `backup list`, `backup restore`, and `backup diff` discover backups only from manifests rendered for the selected app IDs. Backup ownership must be derived from rendered app metadata (`appIds`, `appNamespace`, rendered target resources) rather than file paths. If `backup create` matches a secret whose target namespace does not belong to the selected app, the command fails instead of writing a misplaced backup. If `backup restore` encounters such an app-foreign backup, it warns and reports `skipped` instead of applying it. Namespace allowlists and `--all` are not part of backup selection anymore; app IDs are the only scope boundary, and restore applies only the ownership-valid subset of the selected backups to their declared namespaces. Backup SopsSecrets are identified by `hydra-gitops.org/hydra-backup: "true"` annotation and always use `suspend: true`. For `backup diff`, the comparison ignores the same managed annotation prefixes already filtered during backup creation (`kubectl.kubernetes.io/`, `argocd.argoproj.io/`, `helm.sh/`) and removes `metadata.annotations` entirely if that block becomes empty. Bootstrap and uninstall skip backup SopsSecrets; `backup` tag implies uninstall
- **Uninstall via Ref Tags** - Uses `uninstall`, `uninstall-safe`, and `uninstall-force` tags to control deletion behavior; ref-ownership checks include **`runtime`** on uninstall/uninstall-force groups only for **cluster-only** ids (no standalone template render): **`hydra gitops review`** live pass and **`AssignClusterEntitiesToAtMostOneAppByRefs`** use non-runtime predicates for template-mapped ids and full predicates for cluster-only ids; **`hydra local review`** and the template-id pass of **`hydra gitops review`** always omit **`runtime`**-tagged groups
- **Cluster defaults presets (`global.hydra.presets`)** - Builtin embedded YAML plus merged Helm/ConfigMap overrides define **`cel`** and/or **`ids`** predicate groups (`coredns`, `kubernetes`, `flannel`, `canal`, `kubermatic`, `syseleven`, `metakube`, `syseleven-node-problem-detector`, `quobyte`, `cloudinit`, `cinder`), optional **`activates`** (other preset ids to force-enable when a preset is on), and per-predicate merge rules. CEL is evaluated with **`cel.NewEnvWithEntityInventory(renderedAllApps)`**; explicit **`ids`** drive the blanket bootstrap audit expected-id union and appear in **`hydra gitops system`**. They filter **warn-listed** uninstall-force leftovers, add **review** findings when a cluster-only resource both matches a preset and has a Hydra app assignment, suppress **unassigned** ref-ownership findings when a preset matches alone, and support **`hydra gitops system`** diagnostics. See [uninstall-selection.md](details/commands/deletion-and-topology/uninstall-selection.md#global-hydra-presets--builtin-cel-cluster-defaults)
- **Selection/Marking** - Marks entities for deletion/uninstall by app ID, root app, or cluster scope
- **Deletion** - Three-phase deletion: webhook deletion, scale-down with polling, then foreground deletion in topological order
- **Custom Scale Workloads** - All operator CR scale targets are defined exclusively via `global.hydra.scale` in the app's Helm values; there are no built-in defaults in Go code. `ScaleWorkloadMap` (formerly `MergedScaleWorkloadMap`) loads the map from values without any default merging. The generic StatefulSet-skip logic (`ShouldSkipOperatorManagedStatefulSet`) remains: when a StatefulSet's `ownerReference` points to a CR whose GVK is present in the scale map, Hydra skips the StatefulSet and scales the CR instead. The internal helper `shouldSkipOperatorStatefulSet` wraps this check: it first inspects the primary unstructured (template view), then falls back to the live cluster view (`KeyClusterEntity`) for `ownerReferences` which are only set at runtime by the Kubernetes API server. This fallback is essential because `CollectScaleTargets` and `ZeroWorkloads` use `KeyTemplateEntity` as the primary key, and rendered templates never carry `ownerReferences`. Without the fallback, a StatefulSet managed by an operator CR (e.g. Prometheus) would be scaled directly instead of through its CR, causing the operator to fight back and continuously recreate pods. The scale map is used for `hydra gitops scale`, for `hydra gitops apply` (scale-up and zero-replica apply phases), and for the scale-down step at the start of `hydra gitops uninstall`
- **Cluster scale status (read-only)** - `hydra gitops scale status <appId...>` renders the selected apps, merges with a full live cluster list (`ListClusterAll`), resolves refs, and builds a report: for each scale target from `CollectScaleTargets` (built-in workloads plus `global.hydra.scale` CRs), classify **sync** live state as `up` (matches rendered template scale fields), `down` (matches Hydra’s scaled-down shape: zero replicas / suspended job / DaemonSet disabled via the Hydra nodeSelector / custom replica paths all zero), or `out_of_sync` (neither, or missing live object). Template match wins when both apply (e.g. template replicas are zero). **Additionally**, for each **root** workload row and each **dependency** child, when a matching `global.hydra.ready` rule exists (including **built-in default** rules), classify **ready** state as `ready` or `not_ready` from the rule’s `cel` list (**all** expressions must **pass**: `null` / `""` / `[]` or omit semantics per [details/cel.md](details/cel.md#global-hydra-ready-rules-predicate-and-cel-list); **any** non-empty string, non-empty list of strings, or evaluation error → `not_ready`). **`readyMessages`** carry those reasons when not ready (list results are flattened). Entities with **no** matching ready rule omit ready fields entirely (not listed in ready-oriented dependency output). **Ready dependencies** are discovered **transitively** along the ref graph in the same spirit as `hydra gitops inspect` / `hydra local inspect` (full ref universe through non-workload hops, not only direct workload→workload edges), but **only** entities that have a matching ready rule participate in the ready-specific dependency listing. Direct **sync** dependencies remain **workload-to-workload** via `ResolveTransitiveWorkloadDeps` as today; each dependency repeats the same **sync** classification and records `optional: true` when the edge carries `optional:startup`. **Default stdout** is human-oriented colored text; **`--yaml`** switches to YAML, with **`--color` / TTY auto** for YAML highlighting. Core reporting remains in `scale_workload_status.go` (or successor helpers); extend tests for dual-dimension output, ready omission, and transitive ready deps. **`--all` / `-A`** disables **`FilterClusterScaleStatusReportOmitFullyHealthyApps`**, which otherwise drops each scale-target row that is fully healthy (**`up`**/**`ok`** + dependencies) or a **missing** Job with at least one dependency and all dependencies **`up`**+**`ready`**
- **`global.hydra.ready` and scale up** - After patch ordering from `TopologicalExecute`, **scale up** waits for dependencies using the **same** ready evaluation: any workload or transitive dependency that has a matching ready rule must satisfy **ready** (all `cel` entries pass per the string / list / null contract) before dependents proceed; any failing check **blocks** the same way a failed replica wait would. Align with **optional:startup**: optional edges continue to delay optional-only workloads to the end of the plan; ready gating applies within those phases to entities that **have** a ready rule. Entities **without** a matching ready rule are not ready-gated (existing built-in `waitReady` / replica / `statusReadyPath` polling remains). See [cluster-scale-flow.md](details/commands/deletion-and-topology/topology-and-scaling/cluster-scale-flow.md#scale-up-data-flow) and [values.md](details/values.md) (`global.hydra.ready`)
- **Topological Execution** - Dependency-aware ordered execution of scale-up/scale-down via `TopologicalExecute`
- **Cluster scale and Pods** - `hydra gitops scale down` runs template workload scale-down (`ScaleDownWorkloads`), then **`ScaleDownClusterOnlyWorkloads`**: built-in workloads that exist only on the cluster (no `KeyTemplateEntity`) but are linked to rendered resources via transitive `ownerReferences` (same association rule as app-associated Pods) are scaled to zero (replicas or DaemonSet disable). Without `--force-scale-down`, Hydra waits up to `--cluster-workload-timeout` (default `1m`) for pods whose **direct** `ownerReferences` point at those workloads to terminate; if any remain, it logs **WARN** and aborts with `ErrClusterWorkloadWaitTimeout`. With `--force-scale-down`, cluster-only patches run with **no** post-patch wait (`--cluster-workload-timeout` is mutually exclusive with `--force-scale-down` on the CLI). Then `ReconcileScaleDownPods` lists `v1/Pod`, merges into entities, and reconciles: template-direct Pods are deleted; app-associated Pods (`ownerReferences`, transitive to template entities) log **WARN** when still present or **Terminating**, or are force-deleted with `--force-scale-down` (`GracePeriodSeconds: 0`). Reconcile also runs when workloads did not mutate but app-associated Pods still need attention. Scale **up** creates missing **template-direct** Pods only; cluster-only workloads are expected to be recreated by controllers when template entities scale back up. Scope remains **`hydra gitops scale` only** for this Pod hook. Full flow: [cluster-scale-flow.md](details/commands/deletion-and-topology/topology-and-scaling/cluster-scale-flow.md#cluster-only-workload-scale-down)
- **Apply-integrated backup restore** - `cluster apply` runs the restore phase only when `--backup-restore` is set and `--skip-backup-restore` is not. It restores backup candidates only from manifests rendered for the selected app IDs. The restore step is a normal apply phase that can report `skipped`, `next`, or `aborted`. Restore logging should use the `Backup overview` as the authoritative summary and must not emit extra per-secret `up-to-date` lines before that overview. Namespace metadata remains relevant for validating backup ownership and target placement, but file paths are never an ownership signal
- **Bootstrap apply vs backup restore (same `v1/Secret` id)** - Duplicate Kubernetes object ids in the template render still deduplicate with a **warn** (`entity.NewEntities`). When **`--sops-decode`** and the integrated backup restore phase are both active (`--backup-restore` and not `--skip-backup-restore`), a `v1/Secret` id that would be **restored from backup** and also appears in the **post-render apply set** (plain `Secret` manifests and/or Secrets derived from non-backup `SopsSecret` conversion) is **ambiguous**; the apply preparation step must **abort with `ErrAborted`** and list both sources. If backup restore is skipped or there are no restore candidates, this check does not apply
- **Automatically numbered apply phases** - `cluster apply` builds its phase list through shared phase types in `hydra-go/base/...`; numbering comes from phase order, not hard-coded literals in the action handler. Before mutating phases, apply classifies rendered vs live resources into **new**, **update**, **replace** (delete-before-apply), **unchanged**, and **delete** (orphans) using one server-side apply dry-run over existing objects; phase handlers filter that classification by resource kind (CRDs, namespaces, webhooks, remaining workload). Phase logs include stable phase ids (for example `apply-crds`) alongside `phase x/y`
- **Apply vs diff** - `hydra gitops diff` prints unified diffs for review; `hydra gitops apply` applies desired state and does not emit a diff, though it may reuse SSA dry-run internals for classification
- **Cluster apply resource filters** - `hydra gitops apply` accepts the same repeatable CEL `--include` / `--exclude` flags as `hydra gitops diff` (via `DefineFlags` / `PredicatesFlag`). After `ApplyTemplatePatchesToEntities`, the rendered manifest set is filtered with the same CEL evaluation model as diff (`filterDiffEntitiesByPredicate` / `MissingKeysReject`). Live inventory uses `ListClusterStrictPredicate` (`core/commands/list_cluster.go`), which **does not** fall back to listing the whole cluster when no API resource type matches at aggregate level (contrast `ListClusterPredicate` / `ListCluster`). Orphan detection is skipped when predicates are set (empty orphan set passed into `ClassifyApplyOperations`). `ClassifyApplyOperations` / `ServerSideDryRunApplyEntities` therefore run on the filtered template and filtered cluster slices only. Validation rejects combining predicates with **`--replace`** or **`--orphan-scale-down`** (including when **`--orphan-scale-down`** is implied by **`--bootstrap`** unless the user passes **`--no-orphan-scale-down`** or sets **`--orphan-scale-down=false`** on the CLI). `FilterApplyOperationsByResourcePredicate` remains available for tests and any future call sites that must post-filter an `ApplyOperations` value. Wiring: `prepareApplyState` (`cli/action/cluster_apply_plan.go`), `validateClusterApplyFlags` (`cli/cmd/cluster_apply_flags_resolve.go`)
- **Template patches (`global.hydra.templatePatches`)** — After Helm render and optional `global.hydra.clones` materialization, Hydra runs a final pass over the manifest set used by `hydra local template` (including clone appendix), `hydra gitops apply`, and `hydra gitops diff`. Rule collection uses **Helm `global.hydra` per selected app** merged with the **union of chart-scoped Hydra ConfigMap `data.hydra` documents** (stable id order), matching how **`hydraAppMergedValuesMap`** merges ConfigMap fragments for effective values: a Hydra config ConfigMap rendered by one app (for example the Argo CD chart) contributes `templatePatches` to **every** selected app’s merged rule set. **`PartitionHydraConfigDocumentsByApp`** assigns ConfigMap documents using **all** cluster `AppId`s (not only the selected subset) so per-app buckets and the union stay consistent. When the Helm template render covers only a **subset** of apps, Hydra augments the document set used for that partition by **entity id** (see `mergeRenderedForHydraPartition` in `core/hydra/template_patch_merge.go`): `hydra local template` passes the existing full-cluster **scope-catalog** render together with the per-app partition render; `hydra gitops apply` and `hydra gitops diff` run an additional **`RenderClusterSelectedApps` over all cluster apps** when the resolved selection is not the full app set, so cross-app carriers are still visible. Each named rule has a CEL **`predicate`** and **`patches`** with mikefarah/yq expressions. A rule with an empty **declaring app** (from a global ConfigMap fragment with empty template `appIds`) applies to every entity; otherwise it applies only when the entity’s primary **`AppIds()[0]`** matches that app. After all matching yq steps, Hydra aborts if **apiVersion**, **kind**, **metadata.name**, or **metadata.namespace** would change. For **v1/ConfigMap** objects that qualify as Hydra config carriers (`hydra-gitops.org/hydra-config: "true"` with non-empty **`data.hydra`**), any YAML change at all is rejected (no-op matches are allowed). **`RenderCluster`** itself stays unpatched so downstream readers of rendered Hydra ConfigMaps still see the pre-patch documents when collecting merge fragments; patch entries are resolved from that unpatch snapshot, then the working apply/template set is mutated. Implementation: `HydraTemplatePatchRuleEntries`, `NewTemplatePatchPipeline`, `ApplyTemplatePatchesToEntities` in `core/commands/template_patch.go`; `ApplyTemplatePatchesUsingPartitionRender` for `hydra local template` (catalog + partition render). **`hydra gitops template`** supplies both the selected-app render and the full-cluster catalog (`RenderClusterAllApps`) when resolving rules, while still applying patches only to each printed app’s entities after `NormalizeApiVersions`
- **Apply log consolidation** - phase logging for `cluster apply` stays phase-oriented: one apply operation should not produce multiple generic `applying N resources` messages for the same resource set
- **Single apply phase plan** - `cluster apply` always builds the same automatically numbered phase list; optional steps are gated by `ClusterApplyBehaviorFlags` (`--sops-decode`, `--down-scaled`, `--scale-up`, `--orphan-scale-down`, `--sync`, `--bootstrap-guard`, `--bootstrap-clones`, `--backup-restore`, `--disable-webhooks`). **`--bootstrap`** implies those behaviors unless the user opts out with the corresponding **`--no-*`** flag; with **`--bootstrap`**, the positive optional flags must not be set explicitly on the CLI (except **`--sync`**, which may be set together with **`--bootstrap`**). When **`--sops-decode`** and **`--backup-restore`** are both active, out-of-scope backup `SopsSecret` resources are removed from the later main apply set
- **Bootstrap guard (cluster apply)** - After `RenderCluster` and CRD eligibility validation, and **before** optional `ConvertSopsSecretsToSecrets`, apply evaluates CEL predicates from `global.hydra.refs` groups tagged `bootstrap-guard` when **`--bootstrap-guard`** is set (or implied by `--bootstrap`). If any entity matches and neither `--bootstrap` nor `--skip-bootstrap-guard` is set, apply aborts with an error listing matching resource IDs. `--bootstrap` and `--skip-bootstrap-guard` are mutually exclusive, as are `--bootstrap-guard` and `--skip-bootstrap-guard`. If `--skip-bootstrap-guard` is set but no predicate matches, apply logs a warning. Apps declare guarded resources via Helm (for example the sops-secrets-operator image-pull `SopsSecret`); the mechanism is generic so additional chicken-and-egg resources can be tagged without new Go special cases
- **Resource cloning (`global.hydra.clones`)** - After optional SOPS decode, Hydra materializes additional template entities from Helm and Hydra ConfigMap rules: CEL predicates select source objects, targets are namespace lists or CEL (same CEL environment as refs, including **`managedNamespaces()`** / **`templateEntities()`** / **`clusterEntities()`** / **`entities()`**). Rendered resources override clones with the same id (`real > clone`); two clone rules producing the same target id fail. `SopsSecret` sources are rejected (SOPS MAC is bound to namespace). Target namespaces are owned via a namespace→`appId` map using three strategies in priority order: declared ownership via `global.hydra.ownerNamespaces` (highest), explicit `v1/Namespace` object owner, fallback unique app in namespace (evaluated with `GroupNamespacesByApp` on a **full-cluster** render for clone materialization). **Kubernetes system namespaces** (`kube-system`, `kube-public`, `kube-node-lease`, and any name with prefix `kube-`) are excluded from all strategies so they are never treated as app-owned for clone resolution and do not trigger ambiguous-owner errors when multiple apps deploy there. **`hydra local config`** merges implicit sole-namespace ownership into the printed `ownerNamespaces` list for the selected app. Clone-derived objects carry `hydra-gitops.org/hydra-clone-source`. Uninstall expands clone targets via `ExpandClonesForUninstall` (optional `--bootstrap` on uninstall to include bootstrap-tagged rules). See [details/clones.md](details/clones.md).
- **Required vs optional scale-up** - startup ordering no longer relies on app-wide workload fanout. Required dependencies are started first; workloads that are only linked through optional refs are delayed until the end of the scale-up phase. Readiness waiting is conditional: workloads that already report the desired ready count must not log `waiting for X to become ready` and must not enter unnecessary wait loops **for built-in replica-style polling**; **`global.hydra.ready`** adds an additional gate where rules match (see **`global.hydra.ready` and scale up** above). DaemonSet restore is authoritative: after scale-down injects `hydra-gitops.org/hydra-disabled=true` into `spec.template.spec.nodeSelector`, scale-up must restore the full rendered-template `nodeSelector` with replace semantics so `hydra-gitops.org/hydra-disabled` is removed whenever the template does not contain it
- **ServerSideApply annotations** - Tracks apply metadata for conflict resolution
- **Webhook resolution & smart disable** - Identifies webhook-backing services; disables non-ready webhooks by patching `failurePolicy` to `Ignore` before apply, restored via SSA in the webhook apply phase

## Test Coverage

Helpers must compile with **`cel.NewEnvWithEntityInventory(rendered)`** (same render as predicate collection) when app-defined predicates may call **`managedNamespaces()`** or other inventory helpers.   - **`core/commands`:** Extend `mark_as_selected_test.go` (and backup/bootstrap helpers) so each scope path that compiles app-defined predicates with namespace inventory uses **`NewEnvWithEntityInventory`** from the **same** render as predicate collection; assert correct selection when predicates filter by **`managedNamespaces()`** or **`templateEntities({"namespace": ns})`**, including backup predicates merged with uninstall marking where applicable.
  - These tests verify the API contract in [CEL details — Entity inventory and managed namespaces](details/cel.md#entity-inventory-and-managed-namespaces-uninstall--backup--refs--clones) and [uninstall selection — Uninstall predicate compilation scope](details/commands/deletion-and-topology/uninstall-selection.md#uninstall-predicate-compilation-scope), not user-visible CLI behavior.
