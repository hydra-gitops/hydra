# Reference Discovery Architecture

References represent dependencies between Kubernetes entities (for example a Deployment referencing a ServiceAccount, a Service selecting a Deployment). Hydra discovers references automatically using CEL-based ref-parsers that analyze entity specifications and extract relationship endpoints.

## Key Concepts

- **Semantic split** — **`label`** is for **graph display** (relationship/target role); **`tag`** drives **behavior** (uninstall, optional refs, startup); **`attributes`** carry **structured** provenance and policy (`origin:*`, `key`, …); **`reverse`** does not change stored **`From`/`To`** but affects **visual** direction and **logical** dependency interpretation for consumers (ordering, grouping, graph). Prefer **YAML** for constant `label` / `tag` / `attributes` / `reverse`; use CEL when values **vary per ref**. **`optional:ref`** is the usual exception (branch-specific in CEL). See [details/references.md](details/references.md#semantic-roles-of-labels-tags-attributes-and-reverse). The built-in **`batch/v1/Job`** parser emits a **`source`** edge to **`v1/Secret/&lt;ns&gt;/argocd-redis`** when a container or initContainer **`command`** includes **`redis-initial-password`** (ArgoCD Helm redis secret init Job), so the graph shows the Job as provisioning that Secret by convention.
- **Ref** - Edge between two entities with type (`direct` for parser-discovered edges, `indirect` for synthetic ordering edges, **`regarding`** for informational Event→subject edges), labels, tags, optional repeated attributes, and optional reverse flag; runtime/materialization semantics belong in the **`origin:generated`** attribute (`job` or `controller`) and in structured attributes such as `origin:app` / `origin:workload`, not in a separate `runtime` ref type. Built-in Kubernetes parsers emit a **`namespace`** label for namespaced entity → `v1/Namespace` edges, and command logic reuses the same label name (`types.RefLabelNamespace`) for **delete / orphan-delete** topological ordering; see [details/references.md](details/references.md#reference-vocabulary-hydra-go)
- **Events / PodMetrics** — Built-in parsers link `events.k8s.io/v1/Event` subjects (`regarding` / `related`), workload-correlated Events, and **`metrics.k8s.io/v1beta1/PodMetrics`** via `clusterEntities()` where inventory overlay is registered; see [details](details/references.md#events-podmetrics-and-workload-correlated-events)
- **RefDefinition** - Intermediate result from ref-parsers: owner entity, direction (incoming/outgoing), endpoint, label, tags, desc, optional repeated attributes, reverse
- **Ref-parsers** - YAML files with CEL `predicate` (matching) and `pick` entries (each entry has required `cel` extraction plus optional YAML `attributes`, `tag`, `label`, `reverse`); embedded via `//go:embed`
- **Ref attributes** - Structured repeated metadata attached to a relation, for example `key=<name>` for explicitly selected Secret or ConfigMap keys or parser provenance such as `origin:app=<appId>` / `origin:workload=<workloadId>` while keeping the endpoint itself resource-oriented; parser YAML should declare them as an `attributes` array whose items are single-entry key/value objects. For **`origin:generated`**, declare the explicit YAML key **`"origin:generated": job`** / **`"origin:generated": controller`** in (`HydraRefGroup` / `HydraRefParser` attributes) rather than CEL `refBuilder()...attribute("origin:generated", ...)`; split into multiple `ref-parsers` rows when one predicate emits mixed edges (see [values.md](details/values.md)).
- **Resource-level endpoints** - built-in ref-parsers model referenced resource identities (for example Secret or ConfigMap names); explicit per-key selections belong in relation attributes instead of the endpoint identity
- **Resolution algorithm** - Load parsers -> extract definitions -> **normalize `RefDefinition` id endpoints** to the cluster’s preferred API version per group/kind (when a `PreferredVersionMap` is supplied) -> build incoming/outgoing index maps -> connect endpoints -> handle missing entities -> merge and sort; consumers that need materialized targets then recursively apply **`"origin:generated": job`** / **`"origin:generated": controller`** semantics until a fixpoint before validating existence, deriving provenance, or exposing the stabilized graph. The preferred-version map is derived from the same cluster API discovery `ScopeInfoMap` used for entity `NormalizeApiVersions` and is cached on `hydra.Cluster` via `PreferredVersions(compute)` so discovery runs once per operation when callers share a cluster.
- **Review refs validation** - both commands run the same pipeline on **sources** (template entities from the apps selected for the review, after `--exclude-app`) and differ only in how **targets** are supplied: **local** review uses the rendered templates of **all** effectively enabled apps on the same cluster as inferred from the selected `appId` set; **cluster** review uses **all** live entities on that cluster. Apps with `enabled: false` are not rendered and supply **no sources**; they are excluded from the **local** template target set only, while **cluster** review still loads the full live inventory for targets. Findings attach only to sources from the selected apps; cross-app targets are allowed whenever the target ID exists in the target set. `--exclude-app` must not shrink the target entity load; it only limits which apps supply **source** entities. A `Secret`/`ConfigMap` target counts as present if a matching object exists in the target set **or** a **virtual** materialization is reachable through declared refs (attributes **`"origin:generated": job`** or **`"origin:generated": controller`**) after **recursive evaluation to a fixpoint**—see [details/references.md](details/references.md#generated-targets-recursive-evaluation-and-fixpoint); `missing target resource` applies only when neither a direct object nor such a stabilized chain accounts for the reference. They include the target when known, use `missing target resource` for absent target objects, report missing explicitly referenced `Secret`/`ConfigMap` keys, group identical target+message combinations across multiple source sites, keep `envFrom` as an existence-only check because it does not contribute `key` attributes, **skip** refs whose `tags` include `optional:ref` (Kubernetes `optional: true` on env, envFrom, volume, or projected sources) so those edges do not produce findings yet, write findings to standard output as sequential YAML sequence elements after grouping and sorting are complete (without marshaling the full list as one value), honor `ColorFlag` defaults (TTY auto-detect) for YAML, and emit debug logs around `HydraAppRefParsers`, `references.Refs`, key-attribute enrichment, target-key normalization, and grouping or sorting when those stages dominate latency
- **Ref consumers (generic)** - Recursive resolution of **`"origin:generated": job`** / **`"origin:generated": controller`** materializations, virtual-target provenance from ref-parser metadata, and same-namespace `SopsSecret`→`Secret` modeling apply to **every** subsystem that consumes `references.Refs` output (review, apply ordering, UI graph, tests), not only `hydra local review` / `hydra gitops review`
- **SOPS as a generic pattern** - Charts declare `SopsSecret`→`v1/Secret` in the **same namespace** via app ref-parsers (typically **`"origin:generated": controller`**, optional display labels); Hydra treats this as a **generic** producer→Secret edge, not as bespoke Go logic keyed to one chart layout
- **Policy vs graph** - External policy engines (for example Kyverno) that **mirror** or mutate secrets operate on **`v1/Secret`** objects; Hydra’s graph may still include the upstream custom resource for dependencies and review, but mirroring rules target the **Secret**, not the `SopsSecret` CR
- **Namespaced source normalization** - Before reference extraction and validation, rendered entities from the selected-app render path must match the full-cluster render contract: for API namespaced kinds, if `metadata.namespace` is missing on the template `Unstructured`, Hydra sets it to the app namespace (same propagation as `ApplyScopeInfoMap` / `ApplyScopeInfoMaps` inside `RenderCluster`). That must happen for any downstream use that keys entities by `Entity.Id()` or resolves refs by namespace, not only for review-specific code. Without this, IDs can contain an empty namespace segment (for example `networking.k8s.io/v1/Ingress//name`) and ref targets can spuriously appear missing when they exist only under the app namespace
- **Provider endpoints** - `ref("provider", "<name>")` for operator dependencies (CRs or infrastructure resources -> operator workload)
- **App-defined parsers** - Operator charts define their own ref-parsers in `global.hydra.refs` values
- **ConfigMap-defined parsers** - Optional `v1/ConfigMap` objects with annotation `hydra-gitops.org/hydra-config` (truthy string) and a `data.hydra` YAML document whose top-level `refs` map matches the same shape as `global.hydra.refs` groups; **`hydra gitops *`** merges cluster-catalog ConfigMaps into each app’s effective `global.hydra` (same carrier union as values merge), optionally gated by **`scope`** (ordered `include` / `exclude` steps; `values` entries use the same glob rules as CLI app patterns, e.g. `**`, `prod.**`). **`hydra local *`** does **not** apply Hydra ConfigMap `data.hydra`. **`global.hydra.scope` in Helm values is rejected** — define `scope` only in ConfigMap `data.hydra`.
- **Tags** - `uninstall`, `uninstall-safe`, `uninstall-force`, `optional:startup`, and `optional:ref` annotate refs; tags are merged additively from chart groups, YAML `tag` on ref groups / ref-parser rows / `pick` items, and CEL `.tag()` when tags differ per emitted ref. Built-in Kubernetes ref-parsers add `optional:ref` in CEL only on the branch that corresponds to `optional: true` (see [details](details/references.md#ref-parser-format)). `optional:startup` is reserved for startup/operator semantics (scale, topo).
- **Transitive workload deps** - `ResolveTransitiveWorkloadDeps` traces chains through non-workload intermediaries via BFS without falling back to app-wide workload fanout
- **`global.hydra.ready`** - Named Helm values under `global.hydra.ready` attach **CEL `predicate` + `cel[]`** readiness rules to entities using the **same entity variables** as ref-parser predicates and CLI filters (`gvk`, `ns`, `name`, `entity`, …). Used by **`hydra gitops scale status`** (`ready` / `not_ready` plus `readyMessages` when not ready; omitted when no rule matches) and **`hydra gitops scale up`** (topological wait: all matching dependencies must pass **all** `cel` checks). Expressions return **`string`**, **`list(string)`**, or **`null`** only; cluster commands register **`templateEntities()`**, **`clusterEntities()`**, **`entities()`**, **`managedNamespaces()`**, and **`involvedObjectEvents(...)`** over the merged render+live inventory. Defaults and merge semantics: [values.md](details/values.md); CEL binding: [details/cel.md](details/cel.md#global-hydra-ready-rules-predicate-and-cel-list)
- **Required vs optional startup** - startup planning first executes required workload dependencies, then optional-only workloads; missing optional providers are ignored
- **Reverse flag** - Swaps logical dependency direction and visual arrow direction (`From`/`To` fields unchanged); enables chain visualization
- **Storage dependency chains** — Built-in parsers connect workloads via `volumeClaimTemplates`, `PersistentVolumeClaim`, `PersistentVolume`, `StorageClass`, and `CSIDriver`. For **`apps/v1/StatefulSet`**, `volumeClaimTemplates` also emit **expected** edges to per-pod PVCs named `<volumeClaimTemplate.metadata.name>-<statefulSet.metadata.name>-<ordinal>` (label **`volumeClaimTemplate`**, attribute **`"origin:generated": controller`**). Ordinals follow **`spec.replicas`** (default **1** when unset/null) and **`spec.ordinals.start`** (default **0** when unset/null). In **`hydra gitops review`**, the CEL inventory for source parsing is augmented with the **live cluster snapshot** so a second pass can match **real** PVCs only when **`metadata.ownerReferences`** contains a **controller** reference to the same StatefulSet (`apiVersion` **`apps/v1`**, matching name/namespace). Chart-defined provider refs still connect infrastructure drivers to their workloads.
- **Reference vocabulary** — Labels, tags, and attribute type names shipped in **hydra-go** are listed centrally in [details/references.md](details/references.md#reference-vocabulary-hydra-go); generic chart/Helm patterns (without app-specific values) are included there as well
- **Kubernetes owner references** — Embedded parser in `ref-parsers/kubernetes/_all.yaml` emits refs from `metadata.ownerReferences` with labels `controller` / `owner` and attributes `origin:owner` (see [details](details/references.md#kubernetes-owner-references))
- **Rancher objectset owner annotations** — The same `_all.yaml` emits a **child → declared owner** `RefDefinition` with **`reverse: true`** (same semantics as **`metadata.ownerReferences`**): stored **`From`/`To`** stay child → owner, while **logical** edges (transitive refs, scale views, TUI) follow **owner → child** so chains such as **Deployment → Service → svclb DaemonSet** align with scale-down reasoning. GVK values such as `/v1, Kind=Service` are normalized via CEL **`objectsetRioOwnerGvkToHydraGvk`**; label **`objectset-owner`**, attribute **`origin:objectset: rio.cattle.io`** (see [details](details/references.md#rancher-objectset-owner-annotations))
