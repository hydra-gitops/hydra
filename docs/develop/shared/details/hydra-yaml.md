# Hydra YAML File Format

## Overview

The `.hydra.yaml` file is the serialized dependency graph that Hydra generates from rendered Helm templates. It describes Kubernetes entities, their groupings, and the references (edges) between them. The Hydra UI loads this file to visualize the dependency graph.

The file is generated by the Go function `view.RenderDependencies()` and parsed by the TypeScript function `parseHydraYaml()` in the UI.

## Top-Level Structure

```yaml
entities: # List of Kubernetes entities
  - ...
groups: # Groups of related entities (visual clusters)
  - ...
references: # Directed edges between entities
  - ...
```

All three top-level keys are always present. When empty, they are rendered as `[]`.

## Entities

Each entity represents a single Kubernetes resource, identified by its fully qualified ID.

```yaml
entities:
  - id: apps/v1/Deployment/default/demo
    appIds:
      - prod.myapp
    templatePath: my-chart/templates/deployment.yaml
    templateIndex: 1
    manifestPath: prod.myapp/apps/v1/Deployment/default/demo.yaml
  - id: v1/ConfigMap/default/app-config
    appIds:
      - prod.myapp
    templatePath: my-chart/templates/configmaps.yaml
    templateIndex: 1
    manifestPath: prod.myapp/(core)/v1/ConfigMap/default/app-config.yaml
  - id: v1/ConfigMap/default/env-config
    appIds:
      - prod.myapp
    templatePath: my-chart/templates/configmaps.yaml
    templateIndex: 2
    manifestPath: prod.myapp/(core)/v1/ConfigMap/default/env-config.yaml
  - id: v1/Secret/default/external-secret
    tags:
      - app:missing
```

### Entity Fields

| Field           | Type     | Required | Description                                                                                                                                                  |
| --------------- | -------- | -------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `id`            | string   | yes      | Fully qualified entity ID (see ID format below)                                                                                                              |
| `appIds`        | string[] | no       | Hydra app IDs that produced this entity (e.g. `["prod.myapp"]`). Format: `<cluster>.<rootApp>` or `<cluster>.<rootApp>.<childApp>`                           |
| `tags`          | string[] | no       | Metadata tags (e.g. `app:missing` for referenced but absent entities)                                                                                        |
| `templatePath`  | string   | no       | Helm template source file path from the `# Source:` comment                                                                                                  |
| `templateIndex` | int      | no       | 1-based position within the template file, when the template produces multiple entities                                                                      |
| `rbacRules`     | object[] | no       | RBAC rules for `Role`/`ClusterRole` entities (`apiGroups`, `resources`, `verbs`, optional `resourceNames`)                                                   |
| `secretKeys`    | string[] | no       | Secret key names for `Secret` entities (values are never stored)                                                                                             |
| `manifestPath`  | string   | no       | Relative path to the rendered manifest file within the `manifests/` directory of a cluster dump (see [cluster-dump-structure.md](cluster-dump-structure.md)) |

### Why `appIds` is a List

A single Kubernetes entity can be produced by **multiple ArgoCD apps**. For example, a `Namespace` like `kube-system` or a shared CRD may be rendered by several independent apps (e.g. `prod.app1` and `prod.app2`). When this happens, the entity carries all of those app IDs: `appIds: ["prod.app1", "prod.app2"]`.

On the Go side, the rendering loop in `commands.RenderClusterSelectedApps()` iterates over every `appId` and calls `entity.AddAppId(appId)` for each rendered entity. `AddAppId` **appends** to the existing slice rather than overwriting it, so when two apps produce the same entity (matched by ID), the entity accumulates both app IDs.

In contrast, a **chart entry** (`HydraChart`) always has a single `appId` string because each chart entry represents exactly one Helm chart instance within one specific app. The UI groups chart entries with the same name and version via `groupCharts()` and collects their individual `appId` values into a merged `appIds` list (`ChartVersionEntry.appIds`) for display purposes.

Selection and filtering use `HasAny` semantics: an entity matches a filter if **any** of its `appIds` is in the selected set (`entities.SelectByAppIds()` in Go, `matchingAppIds.has()` in TypeScript).

### Entity ID Format

The `id` field encodes group, version, kind, namespace, and name:

```text
group/version/kind/namespace/name    (with group)
version/kind/namespace/name          (core API, no group)
```

Examples:

- `apps/v1/Deployment/default/demo` — Deployment "demo" in namespace "default"
- `v1/ConfigMap/kube-system/coredns` — ConfigMap "coredns" in namespace "kube-system"
- `v1/ServiceAccount/default/my-sa` — ServiceAccount "my-sa" in namespace "default"

### Template Source Fields

The `templatePath` and `templateIndex` fields trace each entity back to its Helm template source file.

**`templatePath`** is the path from the `# Source:` comment that Helm adds to rendered manifests. It identifies which template file produced the entity. Example: `my-chart/templates/deployment.yaml`.

**`templateIndex`** is a 1-based index that identifies the position of the entity within its template file. Rules:

- **Omitted (0):** The entity has no template source (e.g. `app:missing` entities that are referenced but not defined in any template).
- **1:** The entity is the first (or only) YAML document in the template file.
- **2, 3, ...:** The entity is the second, third, etc. YAML document in the same template file (separated by `---`).

Example — a template file `configmaps.yaml` that contains two ConfigMaps:

```yaml
entities:
  - id: v1/ConfigMap/default/app-config
    templatePath: my-chart/templates/configmaps.yaml
    templateIndex: 1
  - id: v1/ConfigMap/default/env-config
    templatePath: my-chart/templates/configmaps.yaml
    templateIndex: 2
```

### Tags

Tags are optional string labels for metadata:

| Tag           | Meaning                                                                         |
| ------------- | ------------------------------------------------------------------------------- |
| `app:missing` | Entity is referenced by another entity but not defined in any rendered template |

## Groups

Groups cluster related entities together for visual grouping in the graph. They are computed automatically based on references and merge rules.

```yaml
groups:
  - name: demo (Deployment)
    ids:
      - apps/v1/Deployment/default/demo
      - v1/Service/default/demo-svc
      - v1/ServiceAccount/default/demo-sa
```

### Group Fields

| Field  | Type     | Required | Description                                          |
| ------ | -------- | -------- | ---------------------------------------------------- |
| `name` | string   | no       | Human-readable group name (e.g. `demo (Deployment)`) |
| `ids`  | string[] | yes      | List of entity IDs that belong to this group         |

Only groups with more than one entity are included. The group name is derived from the seed entity (typically a workload like Deployment or StatefulSet).

## References

References are directed edges between entities, representing Kubernetes relationships such as service selectors, volume mounts, RBAC bindings, etc.

```yaml
references:
  - from: apps/v1/Deployment/default/demo
    to: v1/ServiceAccount/default/demo-sa
    labels:
      - serviceAccount
  - from: rbac.authorization.k8s.io/v1/RoleBinding/default/demo-rb
    to: v1/ServiceAccount/default/demo-sa
    labels:
      - subject
    reverse: true
```

### Reference Fields

| Field     | Type     | Required | Description                                                                                                                                                   |
| --------- | -------- | -------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `from`    | string   | yes      | Source entity ID                                                                                                                                              |
| `to`      | string   | yes      | Target entity ID                                                                                                                                              |
| `labels`  | string[] | no       | Relationship type labels (e.g. `serviceAccount`, `volume`)                                                                                                    |
| `reverse` | bool     | no       | When `true`, `parseHydraYaml()` swaps `from`/`to` at parse time so the data model contains the visual traversal direction. See [Reverse Flag](#reverse-flag). |

### Common Reference Labels

| Label             | Relationship                                   |
| ----------------- | ---------------------------------------------- |
| `serviceAccount`  | Workload → ServiceAccount                      |
| `volume`          | Workload → PVC/ConfigMap/Secret (volume mount) |
| `env`             | Workload → ConfigMap/Secret (env var)          |
| `envFrom`         | Workload → ConfigMap/Secret (envFrom)          |
| `imagePullSecret` | Workload → Secret (image pull)                 |
| `roleRef`         | RoleBinding → Role/ClusterRole                 |
| `subject`         | RoleBinding → ServiceAccount/Group/User        |
| `ingress`         | Ingress → Service                              |
| `service`         | Service → Workload (selector match)            |

### Reverse Flag

The `reverse` flag is used when a reference is stored in the opposite direction of its visual traversal semantics. For example, the RBAC subject reference is stored as `RoleBinding → ServiceAccount` (i.e. `from: rb, to: sa`), even though the visual chain should show `SA → RB`.

Setting `reverse: true` tells the UI's `parseHydraYaml()` to **swap `from`/`to` at parse time** — so the data model contains the visual traversal direction (`from: SA, to: RB`). The `reverse` flag is preserved as metadata. This swap is necessary for the layout algorithm (left-to-right node ordering) and reachability computation (BFS traversal follows the visual chain `Workload → SA → RB → Role`).

See [layout.md](../../hydra-ui/details/layout.md#references) for a detailed explanation of the swap semantics and visual chain example.

## Complete Example

```yaml
entities:
  - id: apps/v1/Deployment/default/demo
    templatePath: my-chart/templates/deployment.yaml
    templateIndex: 1
  - id: rbac.authorization.k8s.io/v1/Role/default/demo-role
    templatePath: my-chart/templates/rbac.yaml
    templateIndex: 1
  - id: rbac.authorization.k8s.io/v1/RoleBinding/default/demo-rb
    templatePath: my-chart/templates/rbac.yaml
    templateIndex: 2
  - id: v1/Service/default/demo-svc
    templatePath: my-chart/templates/service.yaml
    templateIndex: 1
  - id: v1/ServiceAccount/default/demo-sa
    templatePath: my-chart/templates/serviceaccount.yaml
    templateIndex: 1
groups:
  - name: demo (Deployment)
    ids:
      - apps/v1/Deployment/default/demo
      - rbac.authorization.k8s.io/v1/Role/default/demo-role
      - rbac.authorization.k8s.io/v1/RoleBinding/default/demo-rb
      - v1/Service/default/demo-svc
      - v1/ServiceAccount/default/demo-sa
references:
  - from: apps/v1/Deployment/default/demo
    to: v1/ServiceAccount/default/demo-sa
    labels:
      - serviceAccount
  - from: rbac.authorization.k8s.io/v1/RoleBinding/default/demo-rb
    to: rbac.authorization.k8s.io/v1/Role/default/demo-role
    labels:
      - roleRef
  - from: rbac.authorization.k8s.io/v1/RoleBinding/default/demo-rb
    to: v1/ServiceAccount/default/demo-sa
    labels:
      - subject
    reverse: true
  - from: v1/Service/default/demo-svc
    to: apps/v1/Deployment/default/demo
    labels:
      - service
```

## Charts

The optional `charts` section contains Helm chart metadata per app. Each entry maps an `appId` to the chart that was used to render it, including the chart's name, version, and dependencies.

```yaml
charts:
  - appId: prod.myapp.cert-manager
    name: cert-manager
    version: "1.12.3"
    appVersion: "1.12.3"
    dependencies:
      - name: cert-manager
        version: "v1.12.3"
        repository: https://charts.jetstack.io
  - appId: prod.myapp.argocd
    name: argocd
    version: "5.46.0"
```

### Chart Fields

| Field          | Type     | Required | Description                                   |
| -------------- | -------- | -------- | --------------------------------------------- |
| `appId`        | string   | yes      | Hydra app ID (same format as entity `appIds`) |
| `name`         | string   | yes      | Chart name from `Chart.yaml`                  |
| `version`      | string   | yes      | Chart version from `Chart.yaml`               |
| `appVersion`   | string   | no       | Application version from `Chart.yaml`         |
| `dependencies` | object[] | no       | Chart dependencies from `Chart.yaml`          |

### Chart Dependency Fields

| Field        | Type   | Required | Description                                |
| ------------ | ------ | -------- | ------------------------------------------ |
| `name`       | string | yes      | Dependency chart name                      |
| `version`    | string | yes      | Dependency version (may be a SemVer range) |
| `repository` | string | no       | Repository URL or `file://` path           |

The `charts` section is generated by `commands.CollectChartInfo()` and passed to `view.RenderDependencies()`. When no chart info is available, the section is omitted entirely.

## Generation Pipeline

```text
Helm Chart
  │
  ▼
helm.Template()                          Renders chart with values
  │
  ▼
client.Run(chart, values)                Helm engine produces manifest with # Source: comments
  │
  ▼
"---\n# Source: path/to/template.yaml\n..."
  │
  ▼
helm.SplitManifestMap()                  Splits by --- and extracts # Source: paths
  │                                      Returns map[path][]YamlString
  ▼
entity.NewEntitiesFromYaml()             Converts YAML documents to Entity objects
  │                                      Sets templatePath and templateIndex (1-based)
  ▼
view.ToModel()                           Computes groups, references, template source info
  │
  ▼
view.RenderDependencies()                Marshals DependenciesModel to YAML
  │
  ▼
.hydra.yaml
```

## Go Types

The YAML structure is defined by these Go types in `hydra-go/core/view/dependencies.go`:

```go
type DependenciesModel struct {
    Entities   []IdModel    `yaml:"entities"`
    Groups     []GroupModel `yaml:"groups"`
    References []RefModel   `yaml:"references"`
    Charts     []ChartModel `yaml:"charts,omitempty"`
}

type IdModel struct {
    Id            types.Id      `yaml:"id"`
    AppIds        []types.AppId `yaml:"appIds,omitempty"`
    Tags          []string      `yaml:"tags,omitempty"`
    TemplatePath  string        `yaml:"templatePath,omitempty"`
    TemplateIndex int           `yaml:"templateIndex,omitempty"`
    ManifestPath  string        `yaml:"manifestPath,omitempty"`
    RbacRules     []RbacRuleModel `yaml:"rbacRules,omitempty"`
    SecretKeys    []string        `yaml:"secretKeys,omitempty"`
}

type GroupModel struct {
    Name string     `yaml:"name,omitempty"`
    Ids  []types.Id `yaml:"ids"`
}

type RefModel struct {
    From    types.Id `yaml:"from"`
    To      types.Id `yaml:"to"`
    Labels  []string `yaml:"labels,omitempty"`
    Reverse bool     `yaml:"reverse,omitempty"`
}

type ChartModel struct {
    AppId        types.AppId            `yaml:"appId"`
    Name         string                 `yaml:"name"`
    Version      string                 `yaml:"version"`
    AppVersion   string                 `yaml:"appVersion,omitempty"`
    Dependencies []ChartDependencyModel `yaml:"dependencies,omitempty"`
}

type ChartDependencyModel struct {
    Name       string `yaml:"name"`
    Version    string `yaml:"version"`
    Repository string `yaml:"repository,omitempty"`
}
```

## TypeScript Parsing

The UI parses the file in `src/parseHydra.ts` via `parseHydraYaml()`. Entity IDs are split into their components (group, version, kind, namespace, name) by `parseEntityId()`.
